1. Цель работы

Целью лабораторной работы является практическое освоение технологий больших данных путём построения полноценного ETL-конвейера, включающего:

  • загрузку плоских CSV-файлов в реляционную СУБД с помощью Apache Spark;
  • трансформацию «плоских» данных в многомерную модель данных типа «звезда» (Star Schema) в PostgreSQL;
  • формирование шести аналитических витрин данных (Data Marts) в колоночной СУБД ClickHouse.

Дополнительные задачи:

  • разработка Docker Compose-инфраструктуры для воспроизводимого развёртывания стека PostgreSQL + ClickHouse + Spark;
  • решение проблемы пересечения суррогатных идентификаторов при объединении нескольких CSV-файлов с помощью детерминированного хэширования (xxhash64);
  • документирование архитектуры и схемы БД.

2. Описание исходных данных

Исходные данные представлены десятью CSV-файлами (MOCK_DATA.csv, MOCK_DATA (1).csv … MOCK_DATA (9).csv), каждый содержит по 1 000 строк — итого 10 000 записей.

Предметная область — продажи товаров для домашних животных (корм, аксессуары, игрушки и пр.) через сеть магазинов. Каждая строка CSV описывает одну транзакцию продажи и содержит денормализованные сведения о нескольких сущностях.

4. ETL-конвейер (Apache Spark)

Конвейер состоит из трёх последовательно запускаемых Spark-джобов, объединённых скриптом run-pipeline.sh.

4.1 Шаг 0 — загрузка сырых данных (00_load_raw.py)

Spark читает все 10 файлов MOCK_DATA*.csv из директории /data/csv/ единым вызовом с поддержкой многострочных значений и кавычек-экранирования. Все поля приводятся к типу string и записываются в таблицу mock_data в PostgreSQL в режиме overwrite.

Результат: таблица mock_data — 10 000 строк, 49 колонок.

4.2 Шаг 1 — построение модели «звезда» (01_star_schema.py)

Spark читает mock_data из PostgreSQL и выполняет набор трансформаций:

  1. Генерация суррогатных ключей функцией xxhash64 от натуральных атрибутов каждой сущности (см. таблицу 3).
  2. Приведение типов — числовые поля кастуются в int, decimal, double; даты парсятся форматом M/d/yyyy.
  3. Дедупликация измерений — dropDuplicates по суррогатному ключу.
  4. Запись 7 таблиц в PostgreSQL (БД bigdata) в режиме overwrite.

Таблица 3 — Вычисление суррогатных ключей измерений

4.3 Шаг 2 — витрины в ClickHouse (02_clickhouse_reports.py)

Spark читает 5 таблиц из PostgreSQL (факт + 4 измерения), выполняет агрегации через groupBy/agg, присоединяет атрибуты через join и записывает результаты в 6 таблиц ClickHouse.

Особенность: стандартный JDBC-диалект Spark в режиме overwrite выполняет DROP TABLE + CREATE TABLE, что несовместимо с движком MergeTree (требует явного ORDER BY). Решение — перед каждой записью выполняется TRUNCATE через HTTP-интерфейс ClickHouse (порт 8123), а Spark работает в режиме append.

9.2 Аналитические запросы к витринам ClickHouse

-- Топ-10 продуктов по выручке
SELECT product_name, total_revenue, order_count
FROM reports.product_sales_mart
ORDER BY total_revenue DESC LIMIT 10;

-- Топ-10 клиентов по сумме покупок
SELECT customer_name, country, total_spent, avg_check
FROM reports.customer_sales_mart
ORDER BY total_spent DESC LIMIT 10;

-- Помесячные тренды выручки
SELECT year, month, total_revenue, order_count, avg_order_size
FROM reports.time_sales_mart
ORDER BY year, month;

-- Топ-5 магазинов
SELECT store_name, store_city, store_country, total_revenue
FROM reports.store_sales_mart
ORDER BY total_revenue DESC LIMIT 5;

-- Топ-5 поставщиков
SELECT supplier_name, supplier_country, total_revenue, avg_product_price
FROM reports.supplier_sales_mart
ORDER BY total_revenue DESC LIMIT 5;

-- Продукты с наивысшим рейтингом и наибольшим числом отзывов
SELECT product_name, rating, review_count
FROM reports.quality_mart
ORDER BY rating DESC LIMIT 10;

SELECT product_name, review_count, total_revenue
FROM reports.quality_mart
ORDER BY review_count DESC LIMIT 10;

9.3 Запуск конвейера

docker compose up -d                 # поднять инфраструктуру
bash run-pipeline.sh                 # выполнить все 3 Spark-джоба

Для каждого джоба Spark выводит в лог количество записанных строк, что позволяет быстро убедиться в корректности трансформаций без подключения к СУБД.

10. Заключение

В ходе лабораторной работы был разработан и запущен полноценный ETL-конвейер на базе Apache Spark, обрабатывающий 10 000 записей о продажах товаров для домашних животных.

Достигнутые результаты:

  1. Реализована загрузка 10 CSV-файлов в PostgreSQL через Spark JDBC с корректной обработкой многострочных строк и символов экранирования.
  2. Построена многомерная модель «звезда» в PostgreSQL: 1 таблица фактов (fact_sales) и 6 измерений (dim_customer, dim_seller, dim_product, dim_store, dim_supplier, dim_date).
  3. Решена проблема коллизии идентификаторов при объединении 10 файлов: применены детерминированные суррогатные ключи на основе xxhash64 от натуральных атрибутов сущностей.
  4. Сформированы 6 аналитических витрин в ClickHouse с движком MergeTree, обеспечивающим высокую производительность OLAP-запросов.
  5. Разработана воспроизводимая инфраструктура на Docker Compose с тремя сервисами (PostgreSQL, ClickHouse, Spark).

Технологический стек полностью соответствует требованиям к современным Big Data решениям: Spark обеспечивает горизонтально-масштабируемую трансформацию, PostgreSQL — надёжное реляционное хранилище, ClickHouse — колоночное OLAP-хранилище для аналитики в реальном времени.
